## 2026-06-22 — Роль LLM: слоистая модель, не дерево ивентов

Решение: LLM не водит детерминированные расчёты (экономика/население/бой остаются в движке), но явно отвечает за обработку действий игрока, генерацию событий и симуляцию реакции мира — без заранее прописанного дерева ивентов с весами.

Слоистая модель:
- **Числовой слой (движок, детерминированный)** — экономика, население, истощение, производство.
- **Слой решений/нарратива (LLM)** — реакции стран, новые события, дипломатическая переписка, итоговый текст. LLM интерпретирует числа, не считает их.

Связка: движок считает дельты за ход → промт (дельты + действия игрока + краткая история) → LLM выдаёт реакции/события → `LLMResponseValidator` проверяет допустимость → применяется.

Причина: цель — генеративная альтистория, а не Paradox-стратегия с текстовой обёрткой; но детерминизм чисел обязателен для воспроизводимости/баланса.

Расширенные роли LLM (согласованы, не все спроектированы детально): интерпретация свободных действий игрока; генерация событий по порогам состояния; "память" страны между вызовами (краткая сводка позиции — **формат и механизм обновления не спроектированы**); дипломатическая переписка как текст; разведка/слухи как недостоверная информация; условия мира (территории решает движок, формулировки — LLM); периодическая самопроверка согласованности истории.

Ограничители (часть не реализована): лимит на масштаб одного события (нельзя аннексировать державу одним решением); числа всегда у движка; лог решений LLM с привязкой к ходу (**не реализован**); явная инструкция про точку дивергенции от реальной истории в промте; не просить LLM решать за много стран в одном вызове (1-3 "заголовочных" решения, остальное — пороговые правила).

Статус: концептуально решено. `LLMResponseValidator` сейчас проверяет только структуру/существование стран, **не магнитуду последствий** — нужно расширить.

## 2026-06-22 — LLM как "ИИ" стран: без полноценного ИИ

Решение: большая часть поведения стран — не ИИ, а пороговые формулы, одинаковые для всех стран (уже работает: `EconomyTick`/`ResearchTick`/`MilitaryTick`/`PopulationTick`/`DiplomacyTick`). Найдена дыра: `country.economy.militarySpending/researchSpending/...` никогда не меняются для ИИ-стран (бюджет замёрз на значениях из файла сценария, см. `EconomyTick.ts:24-31`).

Решение по дыре: 3-5 простых нудж-правил (не утилити-AI), например "война N ходов подряд → +X% military за счёт welfare", "stability ниже порога → сдвиг к welfare", "стабильный дефицит → урезание расходов". **Конкретные пороги/проценты не определены.**

Тиры стран: топ-15-20 держав получают периодические LLM-решения по качественным вопросам (война/пакты/нац. фокусы), остальные — только пороговые правила, без LLM. **Явный список тиров не зафиксирован** — вероятно совпадает с "high-detail countries" из `AGENTS.md` (USSR/USA/China/Germany/UK/France), требует подтверждения.

Вариативность характера: один раз при создании партии засеять каждой стране 2-3 случайные константы (`aggressiveness`, `riskTolerance`), смещающие пороги правил — дешёвая иллюзия индивидуальности без реального ИИ.

Причина: пользователь явно не хочет писать полноценный утилити-AI/decision-tree; нужен правдоподобный, но дешёвый механизм.

Статус: концептуально решено, не реализовано.

## 2026-06-22 — Threat/Rivalry Response: вызов игроку без полной зависимости от LLM

Решение: чтобы игрок не мог бесконечно "снежным комом" выигрывать без сопротивления, вводится **детерминированный** rubber-band механизм, отдельно от LLM:

1. **Threat/Rivalry score** — разрыв силы (военная+экономическая) между игроком и соперником, по аналогии с `calculateBaseInfluence`/tension-расчётом в `DiplomacyService` (расширить существующее, не писать с нуля). Превышение порога N ходов подряд → гарантированные контрмеры (рост `militarySpending`, коалиция, шпионаж). Решение "нужна ли реакция" — **не LLM**.
2. **Кризис легитимности после войны** — исход войны → детерминированное изменение `PoliticsState.legitimacy/governmentSupport` → провал порога → гарантированный триггер "кризиса лидерства" (само срабатывание не LLM).

LLM отвечает только за **форму**: конкретное название политики/блока/риторику (п.1), конкретный вид кризиса — импичмент/протесты/чистка, в зависимости от `governmentType` (п.2).

Пример из обсуждения: победа в Корейской войне → краткосрочные волнения/импичмент в США (механизм 2) → среднесрочное усиление противодействия игроку (механизм 1). Механизмы независимы, могут идти параллельно.

Причина: пользователь хочет настоящий вызов ("выиграть у умного оппонента"), но доверять баланс LLM целиково — риск (слабые модели, нет гарантии срабатывания). Решение/форма разделены: решение — надёжная математика, форма — генеративный текст.

Статус: концептуально решено. **Не определено**: конкретные формулы/пороги для threatPerception и legitimacy-кризиса. Отмечено дублирование `calculateDiplomaticTension` между `DiplomacyService` и `DiplomacyTick` — консолидировать при реализации.

## 2026-06-22 — LLM-провайдеры: регион Россия/СНГ

Решение: ручной copy/paste (уже описан в `ARCHITECTURE.md`) остаётся основным режимом — это не временная мера до API-интеграции, а осознанно более надёжный вариант для региона (веб-чаты живучее API при нестабильных геоблоках). Для автоматизации — абстракция `LLMProvider` (`complete(prompt): Promise<string>`) с реализациями `ManualClipboardProvider` (дефолт) / `GigaChatProvider` / `OpenRouterProvider` / `OllamaProvider`, переключение через конфиг.

Кандидаты (на момент исследования, июнь 2026, тиры бесплатного доступа нестабильны и могут измениться):
- **GigaChat (Сбер)** — основной кандидат на автоматизацию: хостится в РФ, не подвержен геоблокам, 1M токенов/12 мес бесплатно.
- **OpenRouter (бесплатные модели)** — без карты, но доступность из РФ нестабильна, нужно проверять руками.
- **Ollama локально** — план Б, всегда доступен, слабее качеством.
- Gemini/Groq/Cerebras — американская инфраструктура, риск геоблока (Gemini подтверждённо недоступен в регионе пользователя).
- Автоматизация через скрейпинг веб-чата (Playwright и т.п.) — **решено не делать**: нарушает ToS, банится анти-бот защитой, хрупко.

Статус: решено архитектурно. **Не выполнено**: тестирование GigaChat/OpenRouter/Ollama на реальном промте с JSON-схемой действий — нужно сравнить соответствие схеме, отсутствие выдуманных `targetCountryId`, качество на русском.

## 2026-06-22 — Честная оценка LLM как "оппонента"

Зафиксировано как принятый риск, не решение: LLM (особенно бесплатные модели, доступные в регионе) теряет память между вызовами без специального механизма; деградирует при решении за много сущностей в одном промте; склоняется к реальной истории, если не дать явную инструкцию про точку дивергенции; не гарантирует стратегическую глубину. `LLMResponseValidator` проверяет структуру, не смысл.

Вывод: где нужен гарантированный баланс/вызов — решение должно быть детерминированным (см. Threat/Rivalry Response), LLM — только форма. Пользователь (как ГейМ-мастер/редактор) должен быть готов вручную править нелогичные решения LLM, не доверять автопилоту полностью.

## 2026-06-22 — Найден баг: асимметричный flip-flop союзов в `DiplomacyTick`

При написании тестов на `DiplomacyTick` обнаружено воспроизводимое поведение: если отношения между странами **несимметричны** (A→B высокое, B→A низкое/отсутствует) и A→B пересекает порог `+70`, то в рамках **одного и того же тика** происходит следующее:

1. При обработке A: `addAlly()` поднимает A→B (+20) и B→A (+10, половина дельты — см. `DiplomacyService.changeRelation`).
2. При обработке B (тот же вызов `diplomacyTick`, следующая итерация внешнего цикла): естественное затухание чуть снижает B→A, после чего пороговая проверка видит B→A ниже 30 — и поскольку B уже формально союзник A (его только что добавили на шаге 1), срабатывает `removeAlly()`, который **тут же отменяет** только что созданный союз и накладывает штраф −30 на отношения.

Итог: союз создаётся и распадается в одном тике, если стороны не успели "догнать" друг друга по отношениям. Воспроизведено тестом `DiplomacyTick.test.ts` ("KNOWN BUG: ..."), не исправлено — это поведенческое изменение симуляции, требует отдельного решения (например: симметрично применять `changeRelation` без множителя 0.5 для пары союз/соперник, либо применять пороговые правила к среднему из двух направлений, а не к каждому отдельно).

**Почему это важно для решений выше**: вся модель "ИИ стран без полноценного ИИ" и Threat/Rivalry Response опирается на надёжность пороговых правил `DiplomacyTick`/`DiplomacyService` — этот баг показывает, что текущая реализация порогов не настолько надёжна, как предполагалось. Перед тем как наращивать на ней Threat/Rivalry Response, стоит решить эту асимметрию.

**Исправлено (2026-06-22)**: союз — двусторонний договор по своей природе, а проверка смотрела только на исходящее отношение инициатора. `addAlly()` теперь вызывается только если порог `> 70` пройден **в обе стороны** (`server/src/simulation/diplomacy/DiplomacyTick.ts`) — односторонняя высокая оценка больше не создаёт "союз", который тут же отменяется при обработке второй страны. Соперничество (`addRival`) оставлено односторонним намеренно — угроза/опасение реалистично воспринимается без взаимности, в отличие от союза. Тест переписан из "KNOWN BUG" в обычный регрессионный тест.

**Поправка (2026-06-22)**: второй пункт техдолга ("консолидировать дублирующийся расчёт tension между `DiplomacyService` и `DiplomacyTick`") был основан на находке самого первого прохода субагента в начале этой сессии и не был перепроверен перед тем, как попасть в `docs/TODO.md`. При проверке оказалось, что `calculateDiplomaticTension` (DiplomacyService) и `calculateBaseInfluence` (DiplomacyTick) считают разные вещи (агрегированная напряжённость страны vs влияние одной страны на другую) — дублирования нет. Обе функции не вызываются ни из чего, но это не повод их удалять: они зарезервированы под Threat/Rivalry Response (P3 в `docs/TODO.md`), который планировал именно их расширять.

**Стало**: карта занимает весь `game-content`; вместо двух фиксированных слотов — список независимых окон (`client/src/hooks/useWindows.ts`, тип `WindowInstance[]`). Каждое окно:
- перетаскивается за заголовок (`Window.tsx`, drag через `mousedown`/`mousemove`/`mouseup` на `window`, позиция — state, не CSS-transform);
- по умолчанию занимает размер по содержимому (`width/height: auto`, ограничено `max-width: min(560px, 100vw-32px)` чтобы не разъезжалось бесконтрольно) — это попутно исправило баг из прошлой итерации (таблица мирового рейтинга обрезалась в фиксированных 320px);
- ручка ресайза в углу (`.window-resize-handle`) — при ручном ресайзе верхний лимit снимается (`maxWidth: calc(100vw - 32px)` через inline style, который перебивает CSS-класс), потому что бессмысленно тянуть за угол, если он всё равно упрётся в дефолтный потолок;
- клик по окну поднимает его поверх остальных (`zIndex` через общий счётчик в `useWindows`);
- можно открыть несколько окон одновременно — например, сравнить две страны рядом, или держать бюджет открытым и параллельно смотреть инспектор региона.

Дедупликация по `id` (`country:<id>`, `region:<id>`, либо просто `budget`/`research`/`actions`/`ranking`/`territories`): повторное открытие той же страны/панели не создаёт дубль, а фокусирует существующее окно (`openOrFocus`). Кнопки навигации в шапке (`Территории`/`Мир`/`Действия`) и кликабельные стат-пилюли (ВВП→бюджет, Технологии→исследования) используют `toggle`/`onToggle` — повторный клик закрывает то же окно, а не плодит копии.

Удалены `InspectorPanel`'s собственная обёртка-оверлей и отдельный `DropdownPanel.tsx` — обёртку (заголовок, крестик, drag, resize) теперь единообразно даёт `Window.tsx` для всех типов содержимого.

**Проверено живьём** (превью-браузер, 1600×900): открытие нескольких окон одновременно, перетаскивание (через синтетические `MouseEvent` — обычный `preview_click` не подходит для drag-последовательностей), ресайз с снятием лимита, авто-фокус по клику. Содержимое окон (Бюджет/Действия/Инспектор/Рейтинг/Территории) не менялось — менялась только оболочка.

**Не сделано / открыто**: позиция окна не "прилипает" к границам экрана — если окно открыто у правого края и потом увеличено в ширину, оно может уйти за пределы вьюпорта (надо руками подвинуть). Esc для закрытия окна — не реализован. Сохранение позиций окон между ходами/перезагрузкой — не реализовано (сбрасывается при каждом открытии заново, кроме как в рамках одной игровой сессии пока компонент не перемонтирован).

## 2026-06-23 — Редизайн интерфейса (первая итерация): контекстные панели + верхняя панель статов

Перед оконной моделью выше был промежуточный шаг, тоже стоит зафиксировать. Пользователь показал референс (HOI4/EU4-стиль: верхняя панель статов, вкладки, левая панель страны, плавающие блоки) и явно попросил взять **структуру/плотность, не цвета** — палитра/шрифты (латунь/антрацит/стенсиль/моно из первой итерации дизайна) остались.

Решения:
- Левая и правая колонки перестали быть постоянными — появляются только по клику (этот принцип позже обобщён в оконную модель выше).
- Верхняя панель (`TopStatBar.tsx`) стала плотной: страна+идеология/госустройство, статы (ВВП/Население/Армия/Стабильность/Технологии/Легитимность), штамп статуса, дата/эпоха, кнопка хода. На ширине 1600px это не влезало в одну строку (см. ниже про найденный баг) — сделано двумя явными рядами.
- Добавлен `InspectorPanel.tsx` — обобщённая замена `PlayerCountryPanel`, показывает **любую** страну или регион (не только страну игрока), с кликабельными дипломатическими отношениями (переход на другую страну) — этого не было вообще ни в одной из прошлых версий интерфейса.
- `ResourceTicker.tsx` переделан в компактные "биржевые" чипы (3-буквенный код + сокращённое число, полное название в `title`) вместо полного списка с названиями.
- `WorldRankingPanel.tsx`/`TerritoriesPanel.tsx` выделены в отдельные компоненты, строки кликабельны (открывают инспектор).
- Намеренно не сделано в этой итерации: вкладки Армия/Дипломатия/Политика/Население/История/Статистика с референса — под них нет ни данных, ни экранов; диаграммы по клику на статы — нет хранения истории значений по ходам (отдельная задача, не только UI).

**Найденный баг (исправлен)**: `vite.config.ts` проксировал `/game`, `/scenarios`, `/budget`, `/research`, `/actions` **по префиксу** — это ломало любую статику с похожим именем (`/game_map.json` улетал на Express и получал 404). Пути в proxy переписаны на regex с границей (`^/game($|/)` и т.п.).

**Временное**: `client/src/map/MapView.tsx` точечно использует `/game_map.json` (скопирован в `client/public/`) вместо `/1world-map-full.geojson`, которого ещё нет — явно помечено `// TEMP` в коде, не трогать как постоянное решение, пока пользователь не закончит работу над картой.
