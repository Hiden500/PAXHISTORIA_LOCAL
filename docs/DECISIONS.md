## 2026-06-22 — Роль LLM: слоистая модель, не дерево ивентов

Решение: LLM не водит детерминированные расчёты (экономика/население/бой остаются в движке), но явно отвечает за обработку действий игрока, генерацию событий и симуляцию реакции мира — без заранее прописанного дерева ивентов с весами.

Слоистая модель:
- **Числовой слой (движок, детерминированный)** — экономика, население, истощение, производство.
- **Слой решений/нарратива (LLM)** — реакции стран, новые события, дипломатическая переписка, итоговый текст. LLM интерпретирует числа, не считает их.

Связка: движок считает дельты за ход → промт (дельты + действия игрока + краткая история) → LLM выдаёт реакции/события → `LLMResponseValidator` проверяет допустимость → применяется.

Причина: цель — генеративная альтистория, а не Paradox-стратегия с текстовой обёрткой; но детерминизм чисел обязателен для воспроизводимости/баланса.

Расширенные роли LLM (согласованы, не все спроектированы детально): интерпретация свободных действий игрока; генерация событий по порогам состояния; "память" страны между вызовами (краткая сводка позиции — **формат и механизм обновления не спроектированы**); дипломатическая переписка как текст; разведка/слухи как недостоверная информация; условия мира (территории решает движок, формулировки — LLM); периодическая самопроверка согласованности истории.

Ограничители (часть не реализована): лимит на масштаб одного события (нельзя аннексировать державу одним решением); числа всегда у движка; лог решений LLM с привязкой к ходу (**не реализован**); явная инструкция про точку дивергенции от реальной истории в промте; не просить LLM решать за много стран в одном вызове (1-3 "заголовочных" решения, остальное — пороговые правила).

Статус: концептуально решено. `LLMResponseValidator` сейчас проверяет только структуру/существование стран, **не магнитуду последствий** — нужно расширить.

## 2026-06-22 — LLM как "ИИ" стран: без полноценного ИИ

Решение: большая часть поведения стран — не ИИ, а пороговые формулы, одинаковые для всех стран (уже работает: `EconomyTick`/`ResearchTick`/`MilitaryTick`/`PopulationTick`/`DiplomacyTick`). Найдена дыра: `country.economy.militarySpending/researchSpending/...` никогда не меняются для ИИ-стран (бюджет замёрз на значениях из файла сценария, см. `EconomyTick.ts:24-31`).

Решение по дыре: 3-5 простых нудж-правил (не утилити-AI), например "война N ходов подряд → +X% military за счёт welfare", "stability ниже порога → сдвиг к welfare", "стабильный дефицит → урезание расходов". **Конкретные пороги/проценты не определены.**

Тиры стран: топ-15-20 держав получают периодические LLM-решения по качественным вопросам (война/пакты/нац. фокусы), остальные — только пороговые правила, без LLM. **Явный список тиров не зафиксирован** — вероятно совпадает с "high-detail countries" из `AGENTS.md` (USSR/USA/China/Germany/UK/France), требует подтверждения.

Вариативность характера: один раз при создании партии засеять каждой стране 2-3 случайные константы (`aggressiveness`, `riskTolerance`), смещающие пороги правил — дешёвая иллюзия индивидуальности без реального ИИ.

Причина: пользователь явно не хочет писать полноценный утилити-AI/decision-tree; нужен правдоподобный, но дешёвый механизм.

Статус: концептуально решено, не реализовано.

## 2026-06-22 — Threat/Rivalry Response: вызов игроку без полной зависимости от LLM

Решение: чтобы игрок не мог бесконечно "снежным комом" выигрывать без сопротивления, вводится **детерминированный** rubber-band механизм, отдельно от LLM:

1. **Threat/Rivalry score** — разрыв силы (военная+экономическая) между игроком и соперником, по аналогии с `calculateBaseInfluence`/tension-расчётом в `DiplomacyService` (расширить существующее, не писать с нуля). Превышение порога N ходов подряд → гарантированные контрмеры (рост `militarySpending`, коалиция, шпионаж). Решение "нужна ли реакция" — **не LLM**.
2. **Кризис легитимности после войны** — исход войны → детерминированное изменение `PoliticsState.legitimacy/governmentSupport` → провал порога → гарантированный триггер "кризиса лидерства" (само срабатывание не LLM).

LLM отвечает только за **форму**: конкретное название политики/блока/риторику (п.1), конкретный вид кризиса — импичмент/протесты/чистка, в зависимости от `governmentType` (п.2).

Пример из обсуждения: победа в Корейской войне → краткосрочные волнения/импичмент в США (механизм 2) → среднесрочное усиление противодействия игроку (механизм 1). Механизмы независимы, могут идти параллельно.

Причина: пользователь хочет настоящий вызов ("выиграть у умного оппонента"), но доверять баланс LLM целиково — риск (слабые модели, нет гарантии срабатывания). Решение/форма разделены: решение — надёжная математика, форма — генеративный текст.

Статус: концептуально решено. **Не определено**: конкретные формулы/пороги для threatPerception и legitimacy-кризиса. Отмечено дублирование `calculateDiplomaticTension` между `DiplomacyService` и `DiplomacyTick` — консолидировать при реализации.

## 2026-06-22 — LLM-провайдеры: регион Россия/СНГ

Решение: ручной copy/paste (уже описан в `ARCHITECTURE.md`) остаётся основным режимом — это не временная мера до API-интеграции, а осознанно более надёжный вариант для региона (веб-чаты живучее API при нестабильных геоблоках). Для автоматизации — абстракция `LLMProvider` (`complete(prompt): Promise<string>`) с реализациями `ManualClipboardProvider` (дефолт) / `GigaChatProvider` / `OpenRouterProvider` / `OllamaProvider`, переключение через конфиг.

Кандидаты (на момент исследования, июнь 2026, тиры бесплатного доступа нестабильны и могут измениться):
- **GigaChat (Сбер)** — основной кандидат на автоматизацию: хостится в РФ, не подвержен геоблокам, 1M токенов/12 мес бесплатно.
- **OpenRouter (бесплатные модели)** — без карты, но доступность из РФ нестабильна, нужно проверять руками.
- **Ollama локально** — план Б, всегда доступен, слабее качеством.
- Gemini/Groq/Cerebras — американская инфраструктура, риск геоблока (Gemini подтверждённо недоступен в регионе пользователя).

Статус: решено архитектурно. **Не выполнено**: тестирование GigaChat/OpenRouter/Ollama на реальном промте с JSON-схемой действий — нужно сравнить соответствие схеме, отсутствие выдуманных `targetCountryId`, качество на русском.

## 2026-06-22 — Честная оценка LLM как "оппонента"

Зафиксировано как принятый риск, не решение: LLM (особенно бесплатные модели, доступные в регионе) теряет память между вызовами без специального механизма; деградирует при решении за много сущностей в одном промте; склоняется к реальной истории, если не дать явную инструкцию про точку дивергенции; не гарантирует стратегическую глубину. `LLMResponseValidator` проверяет структуру, не смысл.

Вывод: где нужен гарантированный баланс/вызов — решение должно быть детерминированным (см. Threat/Rivalry Response), LLM — только форма. Пользователь (как ГейМ-мастер/редактор) должен быть готов вручную править нелогичные решения LLM, не доверять автопилоту полностью.

## 2026-06-22 — Найден баг: асимметричный flip-flop союзов в `DiplomacyTick`

При написании тестов на `DiplomacyTick` обнаружено воспроизводимое поведение: если отношения между странами **несимметричны** (A→B высокое, B→A низкое/отсутствует) и A→B пересекает порог `+70`, то в рамках **одного и того же тика** происходит следующее:

1. При обработке A: `addAlly()` поднимает A→B (+20) и B→A (+10, половина дельты — см. `DiplomacyService.changeRelation`).
2. При обработке B (тот же вызов `diplomacyTick`, следующая итерация внешнего цикла): естественное затухание чуть снижает B→A, после чего пороговая проверка видит B→A ниже 30 — и поскольку B уже формально союзник A (его только что добавили на шаге 1), срабатывает `removeAlly()`, который **тут же отменяет** только что созданный союз и накладывает штраф −30 на отношения.

Итог: союз создаётся и распадается в одном тике, если стороны не успели "догнать" друг друга по отношениям. Воспроизведено тестом `DiplomacyTick.test.ts` ("KNOWN BUG: ..."), не исправлено — это поведенческое изменение симуляции, требует отдельного решения (например: симметрично применять `changeRelation` без множителя 0.5 для пары союз/соперник, либо применять пороговые правила к среднему из двух направлений, а не к каждому отдельно).

**Почему это важно для решений выше**: вся модель "ИИ стран без полноценного ИИ" и Threat/Rivalry Response опирается на надёжность пороговых правил `DiplomacyTick`/`DiplomacyService` — этот баг показывает, что текущая реализация порогов не настолько надёжна, как предполагалось. Перед тем как наращивать на ней Threat/Rivalry Response, стоит решить эту асимметрию.

**Исправлено (2026-06-22)**: союз — двусторонний договор по своей природе, а проверка смотрела только на исходящее отношение инициатора. `addAlly()` теперь вызывается только если порог `> 70` пройден **в обе стороны** (`server/src/simulation/diplomacy/DiplomacyTick.ts`) — односторонняя высокая оценка больше не создаёт "союз", который тут же отменяется при обработке второй страны. Соперничество (`addRival`) оставлено односторонним намеренно — угроза/опасение реалистично воспринимается без взаимности, в отличие от союза. Тест переписан из "KNOWN BUG" в обычный регрессионный тест.

**Поправка (2026-06-22)**: второй пункт техдолга ("консолидировать дублирующийся расчёт tension между `DiplomacyService` и `DiplomacyTick`") был основан на находке самого первого прохода субагента в начале этой сессии и не был перепроверен перед тем, как попасть в `docs/TODO.md`. При проверке оказалось, что `calculateDiplomaticTension` (DiplomacyService) и `calculateBaseInfluence` (DiplomacyTick) считают разные вещи (агрегированная напряжённость страны vs влияние одной страны на другую) — дублирования нет. Обе функции не вызываются ни из чего, но это не повод их удалять: они зарезервированы под Threat/Rivalry Response (P3 в `docs/TODO.md`), который планировал именно их расширять.

## 2026-06-23 — Найден и исправлен квирк: `LLMService.generatePrompt()` мутировал порядок `game.countries`

## 2026-06-23 — Экономика: замыкание петли дохода (taxRevenue = ВВП × ставка), добыча — только ResourceTick

Контекст: в `EconomyTick.ts` компоненты дохода (`taxRevenue`/`exportIncome`/`stateEnterpriseIncome`/`otherIncome`) статичны — ВВП растёт (регионы множатся на `growthRate`), а налоги за ним не следуют. Плюс два несвязанных эконом-движка: плоский `economy` на `Country` (в тиках) vs секторный `RegionEconomyService.calculateRegionalProduction/aggregateRegionEconomy` (считает `taxIncome`/`resourceExtraction`/`food`/`industrialOutput`, но не подключён). Добыча ресурсов концептуально задвоена (ResourceTick vs RegionEconomyService).

**Решение A (доход) — выбран вариант "taxRevenue = ВВП × ставка":** при старте партии выводится эффективная ставка `taxRate = taxRevenue / gdp` из данных сценария (на каждую страну), далее каждый тик `taxRevenue = gdp × taxRate`. Не требует новых данных сценария, детерминированно, соответствует "лёгкой экономике" (`AGENTS.md`). Плоская модель остаётся; `RegionEconomyService.calculateRegionalProduction` НЕ подключается (остаётся кандидатом под будущую MapFeature-экономику).

**Решение B (открытый вопрос 4 закрыт) — добычу ресурсов считает ТОЛЬКО `ResourceTick`** (→ `stockpile`). `resourceExtraction` в `RegionEconomyService` — зарезервирован под будущий MapFeature-слой, в общий цикл не включается. Задвоения нет.

Причина: минимум новых сущностей, детерминизм, отсутствие задвоения; разблокирует P2 — ИИ-нудж-бюджеты осмысленны только при живом доходе.

**РЕАЛИЗОВАНО (2026-06-23)**: поле `economy.taxRate?` (`EconomyState`), инициализация в `createGame` (после агрегации), пересчёт в `EconomyTick`, +2 теста. Проверено вживую на сценарии 1946 (`createGame`): `gdp × taxRate` точно восстанавливает стартовый `taxRevenue`.

**Находка при проверке (предсуществующая, не от этой правки)**: только `economy.gdp` на огромном масштабе (~2·10¹¹, перезаписывается агрегацией по формуле региона `pop×dev×infra×1000`); budget-поля (`treasury`/`income`/`expenses`/`budgetBalance`) — порядка 10³ (проверено: USSR treasury 5000, budgetBalance +700) и **внутренне согласованы между собой**. Следствие: `taxRate ~10⁻⁹` корректно мостит `gdp → taxRevenue` (доход пропорционально следует за ВВП — P1 работает). НО коэффициенты **budget↔gdp** в `EconomyTick` (`deficitPenalty = budgetBalance/gdp`, `infrastructureBonus = infraSpending/gdp`, влияние на `inflation`/`unemployment`) ≈ 0 — **расходы и дефицит почти не влияют на рост ВВП/инфляцию**. P2-аустерити (`treasury < 0`) РАБОТАЕТ (treasury мал, budgetBalance его двигает). Нужно решение по масштабу данных, чтобы spending влиял на gdp — отдельная балансовая задача, не чинил здесь.

Финализированная спецификация P1-дохода:
- Новое поле `economy.taxRate: number`, выставляется в `createGame` как `taxRevenue / gdp` (на каждую страну; при `gdp === 0` → 0).
- В начале `EconomyTick` (до расчёта `income`/`budgetBalance`): `economy.taxRevenue = economy.gdp × economy.taxRate`.
- `exportIncome`/`stateEnterpriseIncome`/`otherIncome` — **остаются статичными** в первом проходе (export честно оживёт только с торговлей, P3).
- Добыча ресурсов — без изменений, только `ResourceTick`.
- Спенды (`militarySpending`/...) — НЕ трогаем здесь, это P2 (ИИ-нудж-бюджеты).

Реализация — 3 точки: тип `EconomyState` (поле `taxRate`), `createGame` (инициализация ставки), `EconomyTick` (пересчёт). Тесты `EconomyTick` обновить: проверить, что `taxRevenue` следует за `gdp`.

## 2026-06-23 — ИИ-страны (доступное подмножество): аустерити по дефициту + Threat с полной балансировкой

Контекст: P2 (оживить ИИ) упирается в два блокера — `politics.stability/legitimacy/governmentSupport` **статичны** (в симуляции их только читают `MilitaryTick`/`PopulationTick`, никто не пишет), **состояния войны нет** (в `GameState` нет `wars`). Поэтому war-зависимые правила и кризис легитимности отложены до P3 (war-система + динамика политики). Реализуем только то, что работает на динамичном состоянии. Проверено: все статьи расходов дают эффект (military→`MilitaryTick`, research→`ResearchTick`, education→`PopulationTick`/`ResearchTick`, welfare→`PopulationTick`, infrastructure→`EconomyTick`); `budgetBalance`/`treasury`/`gdp` динамичны.

**Архитектура:** новый детерминированный шаг `aiBehaviorTick(game)` (нужен доступ ко всем странам + `playerCountryId`, как `diplomacyTick`, а НЕ `(country, regions)`). Только для ИИ-стран (`id !== playerCountryId`).

### Правило A — Аустерити по дефициту
- Триггер (ИИ-страна): `budgetBalance < 0` И `treasury < 0`.
- Действие: military/research/education/infrastructure/welfareSpending × **0.95** (−5%/тик).
- Пол: не ниже **50%** стартового значения статьи (снимок в `createGame`, как `taxRate`).
- Само-останавливается при `budgetBalance ≥ 0`.

### Правило B — Threat (полная балансировка)
Метрика доминирования игрока над страной X: `dom = calculateBaseInfluence(player, X)` (зарезервированная функция в `DiplomacyTick.ts`: манпавер-ratio×10 + ВВП-ratio×10 + 5, cap 100; равные ≈25, игрок ×2 ≈45, ×3 ≈65). Порог угрозы **THREAT = 50** (игрок ~×2.5).

Если `dom > THREAT` и X — не союзник игрока, развилка по отношению `rel = X.relations[player]` (дефолт 0):
- **Балансировка** (`rel < 0`):
  - `militarySpending × 1.05` (+5%/тик), потолок **40% дохода**.
  - Контр-блок: для каждой другой угрожаемой (`dom(player,Y) > THREAT`) не-союзной игроку страны Y с `rel_Y < 0`: `X.relations[Y] += 5`/тик → со временем пересекает порог авто-союза (>70 обоюдно) → коалиция формируется существующим механизмом `DiplomacyTick`.
  - Сопротивление: влияние игрока над X не растёт.
- **Бандвагонинг** (`rel ≥ 0`):
  - Военного билд-апа против игрока нет.
  - power→influence: `player.influence[X] += (dom − player.influence[X]) × 0.1`/тик → при влиянии >50 X входит в сферу игрока (существующий порог `DiplomacyTick`).

**Честные ограничения:** союзы/сфера сейчас — ярлыки без механических последствий (нет войны, нет бонусов сферы). Балансировка/бандвагонинг двигают ярлыки (отношения/влияние/сфера/коалиция), но «зубы» появятся только с war-системой и бонусами сферы. Военный ответ — единственная часть с немедленным эффектом (→ `MilitaryTick`). Сложность: power→influence по игроку O(n); контр-блок O(k²) по числу угрожаемых k (обычно мало); all-pairs power→influence НЕ делаем (дорого, не нужно для player-снежка).

**РЕАЛИЗОВАНО (2026-06-23)**: `server/src/simulation/ai/AiBehaviorTick.ts` (Правила A+B), вписан в `SimulationEngine.simulateMonth` после `diplomacyTick`; снимок `economy.spendingFloor` в `createGame`; +11 тестов (`AiBehaviorTick.test.ts`). Проверено: 12 ходов полной симуляции на сценарии 1946 без падений. Числа (THREAT=50, шаги, потолки) вынесены константами в начало файла — тюнингуемые.

Статус: РЕШЕНО + РЕАЛИЗОВАНО (доступное подмножество). Числа тюнингуемые, балансировать на симуляции. **Отложено до P3** (war+динамика политики): нудж «война N ходов→military», «низкая stability→welfare», кризис легитимности после войны, шпионаж/коалиция-как-договор. Список стран-тиров (бывш. вопрос 3) для этого детерминированного подмножества не нужен — тиры только для LLM-решений (отложено).

## 2026-06-26 — Экономика: модель единиц "ВВП-якорь + доли" (Q9 закрыт) + найден и исправлен отдельный баг "рост ВВП мёртв"

Контекст: при выборе трека разработки после P1/P2 выбран Q9 (масштаб
экономики). Диагностика подтвердила и уточнила находку 2026-06-23: данные
12 стран (`server/src/data/countries/*.ts`) пишутся как **абсолютные**
бюджетные числа, внутренне согласованные между собой на масштабе `~10³`
(пример: USSR `taxRevenue: 900` = ровно 1% от авторского `gdp: 90000`), но
`country.economy.gdp` **всегда перезаписывается** агрегацией регионов
(`pop × development × infrastructure × 1000`) на масштаб `~10¹¹` — авторский
`gdp` мёртв с рождения. Отсюда 8 формул вида `spending/gdp` (инфляция,
безработица, `infrastructureBonus`/`deficitPenalty` в `EconomyTick`;
`militarySpendingRatio` в `MilitaryTick`; `education/welfareFactor` в
`PopulationTick`; `educationRatio` в `ResearchTick`) схлопывались в `≈0`.

**Почему не патч одной формулы**: `createCountry` — функция-тождество без
дефолтов, каждая страна авторит все 13 чисел бюджета руками. Пользователь
указал явный масштаб развития — **200+ стран** (все современные для
сценария 2000, ещё больше для Викторианской эпохи) — на котором ручная
синхронизация абсолютных бюджетов с населённо-выведенным ВВП на каждую
страну нежизнеспособна.

**Решение (выбрано пользователем — модель "ВВП-якорь + доли"):** `gdp`
остаётся единственной абсолютной величиной (формула региона не тронута).
Новый тип `EconomyProfile` (`shared/src/types/EconomyProfile.ts`) — авторские
масштаб-свободные доли (`taxRate`, доли расходов по статьям,
`treasuryShare`/`exportShare`/...). Архетип-дефолты по `EconomyType`
(`planned`/`mixed`/`market`, `server/src/data/countries/templates/economyArchetypes.ts`)
дают новой стране осмысленный профиль почти без ручного авторинга.
`createCountry` мержит архетип с оверрайдами и строит нулевой `economy`-
placeholder; `createGame` (после агрегации, когда `gdp` известен) выводит
денежные поля как `profile × gdp`. Следствие: `spending/gdp ≈ taxRate×доля`
— одинаковое отношение для страны любого размера, все 8 формул оживают без
правки самих формул. Подтверждено симуляцией (12 ходов, сценарий 1946):
страны одного архетипа дают идентичный `mil/gdp` независимо от абсолютного
ВВП (разница в десятки раз между США и Китаем/СССР по размеру).

**Зафиксировано пользователем**: текущие 12 стран сценария 1946 — **тестовые
фикстуры**, не reference-данные; конвертированы на доли механически (через
архетип), без исторической калибровки — не ориентироваться на их числа. В
`docs/TODO.md` добавлен пункт про упрощение создания/подключения стран
(архетипы — первый шаг; полный onboarding-пайплайн под 200+ — отдельная
будущая задача). Подробности модели — `docs/ECONOMY.md`, статус
1946/1836/2000 — `docs/SCENARIOS.md`.

**Найден и исправлен отдельный, не связанный с Q9 баг — "рост ВВП мёртв"**:
при верификации (прогон 12 ходов) обнаружилось, что ВВП США **снижался**,
хотя бюджет был в профиците (что математически противоречит формуле роста,
которая не может дать отрицательный множитель). Причина:
`updateAllRegionsAndAggregate` (`shared/src/utils/aggregateCountryData.ts`)
вызывалась **каждый ход** в `SimulationEngine.simulateMonth` и **пересчитывала
`region.gdp` с нуля** по формуле региона — это стирало мультипликативный рост
(`region.gdp *= 1+growthRate+...`), который `EconomyTick` только что применил
в той же итерации. Связь "расходы/дефицит → рост ВВП" была мёртвой
**независимо от масштаба Q9** — потому и не было замечено раньше: коэффициенты
роста умножались на величину, которая немедленно отбрасывалась.

Исправлено: функция разделена на `initializeRegionGdp` (формула региона,
вызывается только один раз в `createGame`, когда `region.gdp` ещё не
накопил историю) и `aggregateAllCountries` (только суммирует текущий
`region.gdp` в `country.economy.gdp`, без пересчёта — вызывается каждый ход
в `SimulationEngine`). После фикса ВВП реально растёт (прогон 12 ходов:
США +11%/год, СССР +20%/год для планового архетипа — правдоподобно для
форсированной индустриализации, не финальный баланс).

**Попутно найден и исправлен независимый id-баг**: `Italy.ts` имел
`id: "Italy"`, но регионы сценария 1946 используют `ownerCountryId: "ITA"` —
несовпадение означало, что Italy **всегда** владела 0 регионами (баг
маскировался тем, что авторский `gdp: 45000` никогда пересчитывался при
0 регионах — после перехода на `economy`-placeholder с нулями это стало
`gdp=0` и вызвало NaN-каскад через деление на ноль). Исправлено
переименованием id в `"ITA"`. Дополнительно: во всех 4 местах деления на
`gdp` (`EconomyTick`/`MilitaryTick`/`PopulationTick`/`ResearchTick`) добавлен
явный guard `gdp > 0` — страна без территории (по любой причине, включая
будущую полную оккупацию в war-системе) теперь экономически инертна за тик,
а не превращается в NaN.

Статус: РЕШЕНО + РЕАЛИЗОВАНО. Числа архетипов и коэффициенты роста
(`EconomyTick.ts`, именованные константы) — тюнингуемые, не финальный
баланс. Верифицировано: typecheck/тесты сервера и клиента зелёные (200
тестов), ад-хок прогон 12 ходов на сценарии 1946 без NaN/взрывов.
