# Симуляция

Все игровые системы работают через тики.

Порядок (актуальный статус — см. `server/src/simulation/SimulationEngine.ts`, обновлять этот список при любом изменении состава тиков):

1. EconomyTick — реализован, подключён
2. ResourceTick — реализован, подключён
3. ResearchTick — реализован, подключён
4. PopulationTick — реализован, подключён
5. MilitaryTick — реализован, подключён
6. DiplomacyTick — реализован, подключён
7. EventTick — не реализован

`ProductionTick` удалён (2026-06-22): его расчёт добычи ресурсов дублировал `ResourceTick` другой формулой, а `taxIncome`/`industrialOutput`/`food` не имели потребителя — см. `docs/DECISIONS.md`. Если понадобится экономика на уровне MapFeature (заводы/шахты), проектировать заново с учётом этого пересечения, не реанимировать старый файл.

После выполнения всех тиков выполняется агрегация данных от регионов к странам.

## Правила тиков симуляции

### Использование регионов

Все тики должны использовать регионы для расчётов:

- **PopulationTick**: рождаемость, смертность рассчитываются на уровне регионов
- **EconomyTick**: рост ВВП рассчитывается на основе развития и инфраструктуры регионов
- **ResourceTick**: добыча ресурсов зависит от инфраструктуры региона
- **ResearchTick**: бонус от исследовательских центров в регионах
- **MilitaryTick**: manpower набирается из населения регионов

### Учёт инфраструктуры

- Чем выше инфраструктура региона, тем выше добыча ресурсов
- Чем выше инфраструктура региона, тем выше рост ВВП
- Инвестиции в инфраструктуру дают бонус к росту экономики

### Детерминированность

- Одинаковые входные данные должны давать одинаковый результат
- Не использовать случайные числа без веской причины (исключение — однократный сид "характера" страны при создании партии, см. раздел "Искусственный интеллект стран")
- Результаты должны быть предсказуемыми и проверяемыми

### LLM и тики

Полные правила взаимодействия LLM с движком — в `docs/LLM_RULES.md`, не дублировать здесь. Короткая выжимка: LLM не считает тики, движок — единственный источник истины.

# Экономика

Экономика должна быть понятной и вычислительно дешёвой. Не использовать сложные экономические модели без необходимости.

Основные показатели:

* GDP
* Treasury
* BudgetIncome
* BudgetExpenses
* Inflation
* Unemployment
* Debt

MapFeature (заводы/шахты/порты) как unit экономики — см. `docs/DECISIONS.md`, раздел про MapFeature-шаблоны (черновик, не финализировано).

# Военная система

Военная система должна использовать:

* manpower
* equipment
* organization
* supply

Не использовать посекундную симуляцию боя. Предпочтительна модель фронтов и операций: бой считается на границах контакта между регионами с разными `ownerCountryId` в состоянии войны, не "по всей карте". Снабжение зависит от инфраструктуры региона и связности с тылом.

# Искусственный интеллект стран

**Это не полноценный utility-AI и не decision-tree.** Решение явно принято в `docs/DECISIONS.md` ("LLM как 'ИИ' стран: без полноценного ИИ") — здесь зафиксирован итоговый дизайн, детали и открытые числа — там.

Модель:

- **Рутинная экономика/военные/демография** — детерминированные тики (`EconomyTick`/`ResearchTick`/`MilitaryTick`/`PopulationTick`), одинаковые для всех стран, без какого-либо "решения".
- **Дипломатический дрейф** — пороговые правила (`DiplomacyTick`): natural decay отношений, авто-союз/авто-соперник по порогам, без LLM и без ИИ.
- **Бюджетные нудж-правила** — небольшой набор условных правил (война → сдвиг к military, низкая stability → сдвиг к welfare, дефицит → урезание расходов), не утилити-AI. Конкретные пороги — открытый вопрос в `docs/DECISIONS.md`.
- **Качественные решения** (война/пакты/нац. фокусы) — только для тира топ-15-20 держав, периодически (не каждый ход), через LLM. Остальные страны — только пороговые правила.
- **Threat/Rivalry Response** — детерминированный rubber-band механизм против "снежного кома" игрока; решение о реакции — не LLM, форма реакции — LLM. Подробности в `docs/DECISIONS.md`.
- **Вариативность характера** — посеянные один раз при создании партии случайные константы (`aggressiveness`, `riskTolerance`), смещающие пороги правил — не реальный ИИ, дешёвая иллюзия индивидуальности.

ИИ не должен действовать случайно вне этого посева — любое действие должно объясняться формулой/порогом/LLM-решением, не голым `Math.random()` в моменте принятия решения.

# Альтернативная история

История не должна иметь жёстких рельс.

Исторические события являются вероятностными. Если условия изменились, историческое событие может не произойти, произойти иначе, или привести к альтернативным последствиям.

Полные правила работы LLM с альтистори (точка дивергенции, не тянуть сюжет к реальной истории) — в `docs/LLM_RULES.md`.
